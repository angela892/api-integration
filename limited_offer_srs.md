# 限量商品訂閱與通知系統 SRS

## 1. 業務目標
- **核心價值**：透過限量折扣商品吸引玩家回訪，並利用 SMS 訂閱機制精準導流。
- **行銷邏輯**：利用「顯示數量 > 實際數量」營造熱銷氣氛（饑餓行銷）。

## 2. 功能規格 (Functional Requirements)

### A. 商城前端 (Player Side)
1. **提前發佈狀態**：
   - 條件：當 `currentTime < 折扣時間`。
   - 表現：商品呈現灰色，顯示上架倒數計時（如：15天後）。
   - 動作：點擊「訂閱通知」，需檢核「是否綁定手機」。未綁定者導向綁定頁面。

2. **購買狀態**：
   - 條件：當 `折扣時間 <= currentTime` 且 `庫存 > 0`。
   - 表現：商品變為彩色，顯示折扣價（原價 * 折扣）。

3. **售罄狀態**：
   - 條件：當 `實際數量 <= 0`。
   - 表現：商品變回灰色，壓上「已售完」浮水印。

### B. 管理後台 (Admin Side)
- **列表欄位**：發佈時間、折扣時間、商品名稱、折扣率、購買進度(Real/Total)、顯示數量、VIP門檻。
- **核心機制**：
  - **顯示數量**：前端看到的庫存（例：1000）。
  - **實際數量**：資料庫真實扣除的庫存（例：10）。
  - **SMS 觸發**：當 `currentTime == 折扣時間`，系統自動對訂閱清單發送簡訊。

## 3. API 需求預定義 (For OpenSpec)
- `POST /subscription/subscribe`: 訂閱商品通知。
- `GET /mall/limited-items`: 取得符合 VIP 等級的商品列表。
- `POST /admin/limited-items`: 新增限量促銷。

## 4. 其他需求

### 4.1 權限與驗證
- 訂閱通知、購買操作需驗證玩家帳號與手機綁定狀態。
- 管理後台僅限授權管理者存取。

### 4.2 推播與通知
- 折扣時間到時應自動推送 SMS 給已訂閱玩家。
- 訂閱成功或失敗需回傳明確資訊。

### 4.3 錯誤處理
- 回傳錯誤場景：未綁定手機、訂閱已存在、商品不存在、商品尚未開賣、商品已售罄。

## 5. 未來擴充建議
- 支援多層級 VIP/會員優惠分級。
- 支援更多通知管道（Email/推播/App 通知）。
- 支援商品庫存與顯示數量動態調整策略。
